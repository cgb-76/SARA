## User Constraints (from CONTEXT.md)

### Locked Decisions

**D-01 — Tag population mechanism**
Tags are populated exclusively via sara-lint D-08. Sara-extract leaves `tags: []` as the default empty field. Tags require whole-wiki context to be coherent; per-artifact assignment at extraction time produces incoherent labels that don't align across pages.

**D-02 — Tag vocabulary: free-form, emergent from whole-wiki analysis**
Tags are free-form strings generated by the LLM, derived from a whole-wiki analysis — not from looking at a single artifact in isolation.

D-08 two-phase LLM process:
1. **Vocabulary derivation pass**: LLM reads all wiki artifact pages and identifies what conceptual themes emerge across the corpus as a whole. Output: a proposed tag vocabulary (e.g. `authentication`, `data-governance`, `q1-planning`, `infrastructure`).
2. **Assignment pass**: For each artifact page, LLM evaluates which of the derived vocabulary tags apply.

**D-03 — Tag format: lowercase kebab-case**
All tags are normalised to lowercase kebab-case (e.g. `authentication`, `data-governance`, `q1-planning`). The LLM is instructed to produce kebab-case tags; D-08 enforces this before writing.

**D-04 — Tag purpose: /sara-query filter axis**
Tags are designed as a filter axis for the future `/sara-query` command. They are visible on each artifact page for human browsing in Obsidian, but their primary long-term value is enabling queries like "show me all decisions tagged `infrastructure`".

**D-05 — D-08 approval UX: vocabulary-first, then batch assign**
D-08 uses a two-step approval flow:
1. Present the derived vocabulary to the user for approval/editing (e.g. "Inferred 8 tags: [list]. Approve, edit, or skip?")
2. After vocabulary is approved, assign tags across all wiki pages and present a summary of assignments (page → tags applied). No per-page approval loop.

**D-06 — D-08 re-run behaviour: full replacement**
Every D-08 run re-derives the tag vocabulary from the current corpus and replaces existing tags on all pages. Tags are never merged with previous assignments.

**D-07 — D-08 trigger: always, including auto-invoke from sara-update**
D-08 runs every time sara-lint is invoked, including the auto-invocation from sara-update. No separate flag or opt-in.

**D-08 — Tag count per artifact**
No explicit limit — the LLM decides how many tags apply based on the artifact's conceptual footprint. The LLM is guided to be precise (avoid over-tagging with weak associations) but is not capped to a fixed number.

### Claude's Discretion

- Context window management for large wikis: how many artifact pages the LLM reads in one vocabulary derivation pass vs whether it works via page summaries. Acceptable approaches: (a) read all pages in full if context allows; (b) read frontmatter + first section of each page for vocabulary derivation, then full-page read for assignment. Pick the approach that works at the wiki sizes SARA is likely to see (tens to low hundreds of pages).
- Ordering of D-08 within the overall lint check sequence (before or after D-02 through D-07).
- Exact format of the vocabulary approval prompt — how to present the derived tags to the user for editing before assignment.
- Whether the assignment summary is shown as a table (artifact ID → tags) or inline per-type.

## Summary

Phase 16 activates the `tags: []` schema field that already exists on all SARA entity pages (REQ, DEC, ACT, RSK — confirmed in sara-init templates and CLAUDE.md schema). The implementation is a single additive check — D-08 — added to the existing sara-lint skill. No other skills are modified.

D-08 is architecturally distinct from D-07 (semantic related[] curation) in one critical way: D-07 operates per-artifact (one finding per uncurated page) and flows through the existing per-finding approval loop. D-08 operates on the entire corpus in a single two-phase pass with a vocabulary-level approval gate before assignment — it cannot be decomposed into individual findings because the vocabulary must exist before any assignment can happen.

The Phase 15 work (D-07 semantic curation, D-06 two-pass, sara-lint as seventh check) established all the infrastructure this phase builds on: the lint check sequence, the Step 3 "collect all findings" model, the AskUserQuestion gate pattern, and the atomic commit mechanism. Phase 16 extends this, but with a different UX model — vocabulary approval then batch write (not the existing per-finding Apply/Skip loop).

**Primary recommendation:** Implement D-08 as a standalone section in sara-lint Steps 3 and 5, outside the per-finding approval loop. The vocabulary derivation and batch assignment are a special-purpose flow that does not fit the individual-finding model.

### Pattern 1: D-08 as a post-loop step in sara-lint (not a per-finding check)

**What:** D-08 runs AFTER the Step 5 per-finding loop, as a separate named step. It is not injected into the per-finding approval loop (unlike D-02 through D-07).

**Why:** The vocabulary approval must happen before any assignment. The existing loop processes one finding at a time; D-08's flow is: derive vocabulary → approve vocabulary → assign to all pages → write all pages → commit. This is a batch operation, not an individual finding. Inserting it into the per-finding loop would break the sequential logic.

**Precedent in codebase:** D-07's repair branch inside Step 5 already demonstrates LLM-per-page inference. D-08 extends this to a two-phase whole-corpus pattern.

**When to use:** Always — D-08 runs every invocation (D-07 locked decision).

### Pattern 2: Vocabulary approval gate (AskUserQuestion)

**What:** Single AskUserQuestion presenting the derived vocabulary list for user confirmation before any writes occur.

**Established pattern source:** Phase 13 per-finding AskUserQuestion with ["Apply", "Skip"] — same tool, different options. D-08 uses ["Approve", "Edit", "Skip"].

**Example prompt structure (Claude's Discretion on exact format):**

```
Derived tag vocabulary from {N} artifact pages:

  authentication, data-governance, infrastructure, q1-2026,
  stakeholder-alignment, compliance

Approve this vocabulary, edit it, or skip tag curation?
```

### Pattern 3: Context window management (20-page threshold from D-07)

**What:** D-07 established the 20-page threshold for full-content vs. summary-only reads. D-08 should apply an analogous strategy.

**Recommended approach (Claude's Discretion):** For vocabulary derivation pass on wikis up to low hundreds of pages (SARA's target scale), read frontmatter (`id`, `title`, `summary`) plus the first substantive body section of each page. This is richer than D-07's frontmatter-only summary path while still bounding context usage. For assignment pass: per-page full read is safe because the vocabulary is already derived (the LLM processes one page at a time).

**Rationale:** The vocabulary derivation pass needs enough signal to identify conceptual themes, but does NOT need full body prose of every page. The `summary` field is specifically designed for this purpose (50-word max, per sara-init). For assignment, the LLM needs the full artifact context to accurately determine which vocabulary tags apply.

### Pattern 4: Atomic commit for tag writes

**What:** One git commit for all tag updates from a D-08 run (not one commit per page).

**Why:** D-08 is a full-replacement corpus-wide operation (D-06 locked decision). Committing all at once correctly represents the atomic nature of the vocabulary replacement. The individual per-finding commit pattern (D-02 through D-07) does not apply here.

**Commit message (consistent with existing convention):**
```
fix(wiki): update tags via sara-lint D-08
```

**Stage all modified pages before single commit:**
```bash
git add wiki/requirements/*.md wiki/decisions/*.md wiki/actions/*.md wiki/risks/*.md
git commit -m "fix(wiki): update tags via sara-lint D-08"
echo "EXIT:$?"
```

Note: the notes section of sara-lint already carries "T-13-04 mitigation: commit only stages explicit file paths (never directory globs)" — the D-08 commit should stage explicit paths, not use globs. Build the file list from the write loop.

### Pattern 5: tags: [] field write (frontmatter YAML)

**What:** Replace `tags: []` (or existing `tags: [...]`) in the YAML frontmatter with the new assigned tag list.

**Format options:**
- For empty: `tags: []` (inline)
- For one tag: `tags: [authentication]` (inline acceptable)
- For multiple tags: YAML inline array `tags: [authentication, data-governance, infrastructure]` OR YAML block list — use the same format already present in the file to avoid unnecessary diffs

**Established convention:** sara-init templates use `tags: []` inline. The assignment should replace with an inline YAML array: `tags: [tag1, tag2]`.

**Read-before-write rule:** Must re-read each page immediately before writing (existing sara-lint note: "always re-read a file immediately before writing — another fix in the same loop may have modified it").

### Anti-Patterns to Avoid

- **Inserting D-08 into the per-finding loop:** D-08 is not a "finding" — it is a corpus-level batch operation. If D-08 is modelled as N individual findings (one per page), the vocabulary derivation cannot precede assignment. Keep it as a separate post-loop step.
- **Merging tags from previous runs:** D-06 locks full replacement. Do not union-merge old and new tags.
- **Per-page vocabulary derivation:** The whole point of D-08 is that the vocabulary is derived from the full corpus. Deriving tags per-page defeats the coherence goal.
- **Using Bash text-processing to write frontmatter:** The established rule (in sara-lint notes, CONTEXT.md, and sara-extract) is Read/Write tools only for wiki file modifications — never sed/awk.
- **Committing per page:** The batch commit is the correct model for a full-replacement run. Per-page commits would produce N commits with no individual meaning.
- **Skipping kebab-case normalisation:** The LLM may return tags with spaces or mixed case. D-08 must normalise before writing (strip spaces → hyphens, to lower). This is a code-level normalisation step on the in-memory vocabulary list before any writes.

## Common Pitfalls

### Pitfall 1: D-08 running before per-finding loop completes
**What goes wrong:** If D-08 is positioned in Step 3 as a "check" that produces findings for the per-finding loop in Step 5, the vocabulary derivation happens before assignment can occur — which is correct — but the per-page assignment then gets interleaved with individual D-02–D-07 fixes, breaking the batch model.
**Why it happens:** Mistakenly treating D-08 as equivalent to D-02–D-07 (per-finding checks).
**How to avoid:** Position D-08 as a distinct named step AFTER Step 5 (the per-finding loop), not as a check inside Step 3.
**Warning signs:** If the skill prose says "D-08 findings join {all_findings}", the model is wrong.

### Pitfall 2: Context window overflow on vocabulary derivation pass
**What goes wrong:** Reading all wiki artifact pages in full during vocabulary derivation for a large wiki overflows the context window, truncating pages and producing an incomplete or incoherent vocabulary.
**Why it happens:** Applying the same read strategy used for D-07's per-page repair (small-wiki: full read) to the whole-corpus derivation pass.
**How to avoid:** Use frontmatter + first body section (or `summary` field only) for vocabulary derivation. Full reads are needed only for the per-page assignment pass (one page at a time, manageable context).
**Warning signs:** Vocabulary derivation fails or produces very generic tags when the wiki has more than 30–40 pages.

### Pitfall 3: Forgetting to re-read pages before writing tags
**What goes wrong:** D-08 reads all pages during the derivation pass (Step 6 phase 1), then writes tags after the assignment pass. If D-02–D-07 fixes ran in Step 5, those pages may have changed. Writing stale in-memory content back to disk would undo the Step 5 fixes.
**Why it happens:** Reading pages in the derivation pass and assuming they are still current at write time.
**How to avoid:** Re-read each target page immediately before writing tags (same rule as all other sara-lint writes). The vocabulary and tag assignments computed in the derivation/assignment passes remain valid; only the frontmatter write requires a fresh read.

### Pitfall 4: Kebab-case normalisation missing
**What goes wrong:** LLM returns tags like "Data Governance" or "q1 2026" — stored as-is, breaking grep-ability and future /sara-query filtering.
**Why it happens:** The prompt instructs kebab-case but the enforcement step is omitted.
**How to avoid:** After receiving the LLM vocabulary output, apply in-memory normalisation before presenting to user and before any writes: `lowercase(tag).replace(' ', '-')`. Present the normalised form to the user for approval.

### Pitfall 5: Glob staging in commit violates T-13-04
**What goes wrong:** `git add wiki/**/*.md` stages all files including non-artifact pages (index.md, log.md), or stages unchanged files, creating a noisy commit.
**Why it happens:** Convenience — globbing all wiki dirs is easier than tracking the write list.
**How to avoid:** Build an explicit list of files that were actually written during the assignment pass and stage only those. The sara-lint notes already carry the T-13-04 mitigation directive.

### Pitfall 6: D-08 fires when no artifact pages exist
**What goes wrong:** sara-lint D-08 runs on an empty or near-empty wiki and the LLM produces degenerate output ("no themes found" or hallucinated tags), then prompts the user with a useless vocabulary.
**Why it happens:** No guard clause on the artifact page count before triggering vocabulary derivation.
**How to avoid:** Before running the derivation pass, check the artifact page count. If zero artifact pages found: output "D-08: No artifact pages to analyse — tag curation skipped." and return without prompting.

### wiki/index.md Tags column
