# Phase 16: tagging - Context

**Gathered:** 2026-05-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Activate the existing `tags: []` schema field by implementing sara-lint check D-08 — a whole-wiki LLM pass that derives an emergent tag vocabulary from the full wiki corpus and assigns tags to every artifact page.

Tags are a **concept-level semantic cross-reference** mechanism: distinct from `related[]` (which links specific artifacts to each other), tags link artifacts by shared theme or concept. A tag like `data-governance` may apply to a requirement, a decision, and a risk — connecting them conceptually even if they are not directly related to each other.

No new entity types. No changes to sara-ingest, sara-discuss, sara-extract, sara-update (beyond calling sara-lint), sara-minutes, sara-agenda, sara-init, or the wiki directory structure. The `tags: []` field already exists in all entity schemas — this phase makes it useful.

</domain>

<decisions>
## Implementation Decisions

### D-01 — Tag population mechanism

Tags are populated **exclusively via sara-lint D-08**. Sara-extract leaves `tags: []` as the default empty field (no per-artifact tag inference during extraction). Tags require whole-wiki context to be coherent; per-artifact assignment at extraction time produces incoherent labels that don't align across pages.

### D-02 — Tag vocabulary: free-form, emergent from whole-wiki analysis

Tags are **free-form strings** generated by the LLM, but derived from a whole-wiki analysis — not from looking at a single artifact in isolation.

**D-08 two-phase LLM process:**
1. **Vocabulary derivation pass**: LLM reads all wiki artifact pages and identifies what conceptual themes emerge across the corpus as a whole. Output: a proposed tag vocabulary (e.g. `authentication`, `data-governance`, `q1-planning`, `infrastructure`).
2. **Assignment pass**: For each artifact page, LLM evaluates which of the derived vocabulary tags apply.

This ensures tag labels are consistent across pages — the same concept gets the same tag everywhere, because the vocabulary is derived corpus-wide before any assignment happens.

### D-03 — Tag format: lowercase kebab-case

All tags are normalised to **lowercase kebab-case** (e.g. `authentication`, `data-governance`, `q1-planning`). This ensures consistent querying and grep-ability. The LLM is instructed to produce kebab-case tags; D-08 enforces this before writing.

### D-04 — Tag purpose: /sara-query filter axis

Tags are designed as a **filter axis for the future `/sara-query` command**. They are visible on each artifact page for human browsing in Obsidian, but their primary long-term value is enabling queries like "show me all decisions tagged `infrastructure`". Tag design should anticipate this use.

### D-05 — D-08 approval UX: vocabulary-first, then batch assign

D-08 uses a **two-step approval flow**:
1. Present the derived vocabulary to the user for approval/editing (e.g. "Inferred 8 tags: [list]. Approve, edit, or skip?")
2. After vocabulary is approved, assign tags across all wiki pages and present a summary of assignments (page → tags applied). No per-page approval loop — the vocabulary approval is the human gate.

This is different from D-07 (which uses per-finding approval). D-08's scale (full wiki) makes per-page approval impractical; vocabulary-level approval preserves human oversight while keeping the interaction manageable.

### D-06 — D-08 re-run behaviour: full replacement

Every D-08 run **re-derives the tag vocabulary from the current corpus** and **replaces existing tags** on all pages. Tags are never merged with previous assignments — the current run always produces the canonical tag set for the current state of the wiki. This keeps tags coherent as the wiki grows and new concepts emerge.

### D-07 — D-08 trigger: always, including auto-invoke from sara-update

D-08 runs every time sara-lint is invoked, including the auto-invocation from sara-update. There is no separate flag or opt-in. Tags are re-evaluated on every lint run, keeping them maximally fresh after every ingest cycle.

### D-08 — Tag count per artifact

**No explicit limit** — the LLM decides how many tags apply based on the artifact's conceptual footprint. The LLM is guided to be precise (avoid over-tagging with weak associations) but is not capped to a fixed number.

### Claude's Discretion

- Context window management for large wikis: how many artifact pages the LLM reads in one vocabulary derivation pass vs whether it works via page summaries. Acceptable approaches: (a) read all pages in full if context allows; (b) read frontmatter + first section of each page for vocabulary derivation, then full-page read for assignment. Pick the approach that works at the wiki sizes SARA is likely to see (tens to low hundreds of pages).
- Ordering of D-08 within the overall lint check sequence (before or after D-02 through D-07).
- Exact format of the vocabulary approval prompt — how to present the derived tags to the user for editing before assignment.
- Whether the assignment summary is shown as a table (artifact ID → tags) or inline per-type.

</decisions>

<specifics>
## Specific Ideas

- Tags are a **concept-level** cross-reference (distinct from artifact-level `related[]`). A tag like `data-governance` links a requirement, a decision, and a risk by shared theme — even if none of them directly reference each other. The planner should carry this distinction into the D-08 LLM prompt design.
- The two-phase model (vocabulary derivation → assignment) is the core design insight. The vocabulary derivation pass is what makes whole-wiki coherence possible. Do not collapse this into a single per-page pass.
- Example emergent tags for a typical SARA wiki: `authentication`, `data-governance`, `q1-2026`, `infrastructure`, `stakeholder-alignment`, `compliance`. The LLM should produce labels at this level of abstraction — not overly specific (not `rsk-003-topic`) and not generic (not `important`).
