**Frontmatter:**
```yaml
---
name: sara-ingest
description: "Register a source file in the SARA ingest pipeline, or show pipeline status"
argument-hint: "[<type> <filename>]"
allowed-tools:
  - Read
  - Write
  - Bash
---
```

**Objective:** One short paragraph covering both modes — with arguments: registers a file from `raw/input/` as item N in `pending` stage; without arguments: shows a table of all pipeline items.

**Process — 5 steps:**

**Step 1 — Input validation and mode detection**

Examine `$ARGUMENTS`:

- If `$ARGUMENTS` is empty or blank: execute STATUS mode (Step 5), then STOP.
- If `$ARGUMENTS` contains exactly two words separated by whitespace (`<type> <filename>`): extract `{type}` (first word) and `{filename}` (second word). Proceed to INGEST mode (Step 2).
- Otherwise (wrong number of words): output `"Usage: /sara-ingest <type> <filename>  Valid types: meeting, email, slack, document"` and STOP.

Validate `{type}`: must be exactly one of `meeting`, `email`, `slack`, `document` (case-sensitive). If not: output `"Invalid type '{type}'. Valid types: meeting, email, slack, document"` and STOP. This validation is hardcoded — do not read from config.

Validate `{filename}`: must not contain `/` or `..` (path traversal guard). If it does: output `"Invalid filename '{filename}'. Filename must not contain path separators or '..'."` and STOP.

Validate N argument path (not applicable here — N is generated, not user-supplied for ingest).

**Step 2 — File existence check (INGEST mode)**

Run the following Bash command:
```bash
if [ ! -f "raw/input/{filename}" ]; then
  echo "File not found: raw/input/{filename}"
  echo "Files currently in raw/input/:"
  ls raw/input/ 2>/dev/null || echo "(empty)"
  exit 1
fi
```

If the command exits non-zero: STOP. Do NOT read or modify `pipeline-state.json`.

**Step 3 — Read and update pipeline-state.json (INGEST mode)**

Read `.sara/pipeline-state.json` using the Read tool.

Determine `{type_key}` from `{type}`:
- `meeting` → `MTG`
- `email` → `EML`
- `slack` → `SLK`
- `document` → `DOC`

Increment `counters.ingest.{type_key}` by 1. The new value is `{counter_value}`.

Compute `{new_id}` = `{type_key}` + `-` + zero-padded 3-digit counter value (e.g. counter=1 → `MTG-001`).

Compute `{item_index}` = total number of existing keys in `items` plus 1 (i.e., the next integer index as a string). Example: if `items` currently has keys `"1"` and `"2"`, the new key is `"3"`.

Add new entry to `items` with key `"{item_index}"`:
```json
{
  "id": "{new_id}",
  "type": "{type}",
  "filename": "{filename}",
  "stage": "pending",
  "created": "{today ISO date YYYY-MM-DD}",
  "discussion_notes": "",
  "extraction_plan": []
}
```

Write the modified JSON back to `.sara/pipeline-state.json` using the Write tool. Do NOT use Bash jq, sed, or awk for this write.

**Step 4 — Report success (INGEST mode)**

Output the following (substituting values):
```
{new_id} registered as item {item_index}.
Type: {type}  |  Filename: {filename}  |  Stage: pending
Run /sara-discuss {item_index} to begin the discussion phase.
```

STOP.

**Step 5 — STATUS mode (no-args branch)**

Read `.sara/pipeline-state.json` using the Read tool.

If `items` is empty: output `"No pipeline items registered. Run /sara-ingest <type> <filename> to add one."` and STOP.

Otherwise, output a markdown table:
```
| # | ID      | Type     | Stage     | Filename                 |
|---|---------|----------|-----------|--------------------------|
```
For each key in `items` (sorted numerically by key): append a row:
`| {key} | {item.id} | {item.type} | {item.stage} | {item.filename} |`

STOP.

**Notes block:** Include these gotchas:
- Item keys in `pipeline-state.json items` are string integers ("1", "2"), NOT type-prefixed IDs. Never confuse item key with item.id.
- Filename validation (no `/` or `..`) prevents path traversal — always apply before any file operation.
- Type is hardcoded; do NOT read type list from `.sara/config.json`.
- pipeline-state.json must be read and written atomically using Read + Write — never use jq or sed.
- The item_index counter is NOT the same as the ingest type counter. A project with 2 meetings and 1 email has items "1", "2", "3" but MTG counter=2, EML counter=1.
