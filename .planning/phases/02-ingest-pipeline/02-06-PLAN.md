**Frontmatter:**
```yaml
---
name: sara-update
description: "Execute approved extraction plan â€” write wiki artifacts and commit atomically"
argument-hint: "<N>"
allowed-tools:
  - Read
  - Write
  - Bash

```

Note: AskUserQuestion is NOT needed â€” sara-update is fully automatic. No user choices.

**Objective:** Two sentences: (1) Reads the approved extraction plan from `pipeline-state.json` and writes all wiki artifacts (create or update) plus `wiki/index.md` and `wiki/log.md` in a single atomic git commit. (2) The source file is archived to the type-specific `/raw/` subdirectory in the same commit; stage advances to `complete` only after the commit succeeds.

**Process â€” 5 steps:**

**Step 1 â€” Stage guard and item lookup**

Read `.sara/pipeline-state.json`.

Validate `$ARGUMENTS`: must be a positive integer. If not: output `"Usage: /sara-update <N> where N is a positive integer pipeline item number."` and STOP.

Find item with key `"{N}"` in `items`. If not found: output `"No pipeline item {N} found."` and STOP.

Check `items["{N}"].stage`. Expected: `"approved"`.
If actual stage != `"approved"`:
  Output: `"Item {N} ({id}) is in stage '{actual_stage}'. Run /sara-update N only when stage is 'approved'. Re-run /sara-extract N if you need to revise the plan."`
  STOP.

Store `{item}` = `items["{N}"]`.
Store `{extraction_plan}` = `items["{N}"].extraction_plan`.

If `{extraction_plan}` is empty or null: output `"Extraction plan for item {N} is empty. Re-run /sara-extract {N} to generate an approved plan."` and STOP.

**Step 2 â€” Write wiki artifact files**

Initialize `written_files = []` and `failed_files = []`.

For each artifact in `{extraction_plan}`:

  Determine `{wiki_dir}`:
  - `requirement` â†’ `wiki/requirements/`
  - `decision` â†’ `wiki/decisions/`
  - `action` â†’ `wiki/actions/`
  - `risk` â†’ `wiki/risks/`

  If `artifact.action == "create"`:
    Determine `{entity_type_key}`:
    - `requirement` â†’ `REQ`
    - `decision` â†’ `DEC`
    - `action` â†’ `ACT`
    - `risk` â†’ `RISK`

    Read `.sara/pipeline-state.json` (re-read to get current counter values).
    Increment `counters.entity.{entity_type_key}` by 1.
    Compute `{assigned_id}` = `"{entity_type_key}-"` + zero-padded 3-digit counter (e.g. `REQ-001`).
    Write updated `pipeline-state.json` immediately (counter increment must be persisted before the page write).

    Read `.sara/templates/{artifact.type}.md` to get the template structure.
    Construct the wiki page content by substituting all fields from `artifact` into the template frontmatter and body. Set `source` = `{item.id}`, `schema_version` = `"1.0"`.

    Use the Write tool to create `{wiki_dir}{assigned_id}.md`.
    If write succeeds: append `{wiki_dir}{assigned_id}.md` to `written_files`.
    If write fails: append to `failed_files`. Output partial failure report (see below). STOP.

  If `artifact.action == "update"`:
    Read the existing file `{wiki_dir}{artifact.existing_id}.md`.
    Apply `artifact.change_summary` to the relevant field(s) in the frontmatter or body.
    Use the Write tool to overwrite `{wiki_dir}{artifact.existing_id}.md` with the updated content.
    If write succeeds: append `{wiki_dir}{artifact.existing_id}.md` to `written_files`.
    If write fails: append to `failed_files`. Output partial failure report (see below). STOP.

**Partial failure report format** (if any write fails before commit):
```
## Update Partial Failure

Files written ({count}):
{written_files list}

Files NOT written ({count}):
{failed_files list}

The git commit has NOT been issued. Stage remains 'approved'.
Resolve the write failure, then re-run /sara-update {N}.
Do NOT use git reset â€” no commit was made; the written files are uncommitted changes.
```
STOP after outputting the partial failure report.

**Step 3 â€” Update wiki/index.md and wiki/log.md**

After all artifact files are written successfully:

Read `wiki/index.md`.
For each artifact written:
  - `create`: append a new row â€” `| {assigned_id} | {artifact.title} | open | {artifact.type} | [] | {today YYYY-MM-DD} |`
  - `update`: find and update the existing row matching `{artifact.existing_id}` â€” update `last-updated` to today's date.
Write updated `wiki/index.md`.

Read `wiki/log.md`.
Append the following entry:
`| {item.id} | {today YYYY-MM-DD} | {item.type} | {item.filename} | {list of all artifact IDs written} |`
Write updated `wiki/log.md`.

**Step 4 â€” Archive source file**

Determine `{archive_dir}` from `{item.type}`:
- `meeting` â†’ `raw/meetings/`
- `email` â†’ `raw/emails/`
- `slack` â†’ `raw/slack/`
- `document` â†’ `raw/documents/`

Determine `{archive_prefix}`: zero-padded 3-digit value of `counters.ingest.{type_key}` at the time the item was ingested. This value equals the numeric portion of `{item.id}` (e.g. `MTG-001` â†’ prefix `"001"`).

Compute `{archive_filename}` = `{archive_prefix}-{item.filename}` (e.g. `001-test-fixture.md`).

Check whether the source file is git-tracked:
```bash
git ls-files --error-unmatch raw/input/{item.filename} 2>/dev/null && echo "tracked" || echo "untracked"
```

If tracked: run `git mv raw/input/{item.filename} {archive_dir}{archive_filename}`
If untracked: run `mv raw/input/{item.filename} {archive_dir}{archive_filename}`

**Step 5 â€” Commit, advance stage, and report**

Run the git commit:
```bash
git add wiki/requirements/ wiki/decisions/ wiki/actions/ wiki/risks/ \
        wiki/index.md wiki/log.md \
        .sara/pipeline-state.json \
        {archive_dir}{archive_filename}
git commit -m "feat(sara): ingest {item.id} â€” {item.filename}"
```

Capture the exit code of the `git commit` command.

**If commit SUCCEEDS (exit code 0):**
  Capture `{commit_hash}` via `git log --oneline -1`.
  Read `.sara/pipeline-state.json`.
  Update `items["{N}"].stage` = `"complete"`.
  Write updated `pipeline-state.json`.
  Output:
  ```
  ## Update Complete

  Commit: {commit_hash}
  Artifacts written: {count}
  {written_files list}
  Source archived: {archive_dir}{archive_filename}

  Item {N} ({item.id}) is now complete.
  ```

**If commit FAILS (exit code != 0):**
  Output:
  ```
  ## Update Failed â€” Commit Error

  Files written ({count}):
  {written_files list}

  The git commit failed. Stage remains 'approved'.
  You can re-run /sara-update {N} after resolving the git issue,
  or use `git reset HEAD {written_files}` to undo the uncommitted writes if needed.
  ```
  Do NOT write `stage = "complete"` to `pipeline-state.json`.
  STOP.

**Notes block:**
- CRITICAL: stage advances to "complete" ONLY after the git commit succeeds. Writing stage=complete before commit is a fatal error â€” the item would be permanently stuck (Pitfall 1).
- CRITICAL: entity counter increments happen BEFORE each page write, and the updated counter is written to pipeline-state.json immediately. This prevents duplicate ID assignment if a page write fails and the skill is re-run.
- Source file tracking: if the source was dropped in manually (not git-added), it is untracked. Use `mv` + include both paths in `git add`. If it was committed (e.g. git-tracked from a prior add), use `git mv`.
- Item key vs item ID: the user runs `/sara-update 1`; look up `items["1"]` (not `items["MTG-001"]`). The `item.id` field (MTG-001) is used only in the commit message and artifact source fields.
- schema_version must always be quoted: `"1.0"`.
- `related` fields must use entity IDs only (REQ-NNN, DEC-NNN), never file paths or Obsidian wiki-links.
- Do NOT auto-rollback on partial failure. The user has git history. Report state, let user decide (D-14).
