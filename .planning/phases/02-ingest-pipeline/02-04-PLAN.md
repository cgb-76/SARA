Create `.claude/skills/sara-discuss/SKILL.md` as a new file. Structure: YAML frontmatter → `<objective>` → `<process>` → `<notes>`.

**Frontmatter:**
```yaml
---
name: sara-discuss
description: "Run LLM-driven blocker-clearing session for a pipeline item before extraction"
argument-hint: "<N>"
allowed-tools:
  - Read
  - Write
  - Bash
  - AskUserQuestion
---
```

Note: AskUserQuestion is required because `/sara-add-stakeholder` (invoked inline) uses it for field collection.

**Objective:** Two-three sentences: (1) The skill reads the source document and generates a blocker list of everything that would cause `/sara-extract` to fail or produce wrong output. (2) It works through blockers in priority order — unknown stakeholders first (resolved via `/sara-add-stakeholder`), then ambiguity, context gaps, and cross-link candidates. (3) Done when the blocker list is empty; writes resolved context to `discussion_notes` and advances stage to `extracting`.

**Process — 6 steps:**

**Step 1 — Stage guard and item lookup**

Read `.sara/pipeline-state.json`.

Validate `$ARGUMENTS`: must be a positive integer. If not a positive integer (e.g. empty, non-numeric, zero, negative): output `"Usage: /sara-discuss <N> where N is a positive integer pipeline item number."` and STOP.

Find the item with key `"{N}"` in the `items` object (N is the integer argument — for `/sara-discuss 1`, N = `"1"`).

If no item exists with key `"{N}"`:
  Output: `"No pipeline item {N} found. Run /sara-ingest to register a new item, or run /sara-ingest with no arguments to see the full pipeline status."`
  STOP.

Check `items["{N}"].stage`. Expected stage: `"pending"`.
If actual stage != `"pending"`:
  Output: `"Item {N} ({id}) is currently in stage '{actual_stage}'. Run /sara-discuss N only when stage is 'pending'. If the item is 'extracting', run /sara-extract N."`
  STOP.

Store `{item}` = `items["{N}"]` for use in subsequent steps.

**Step 2 — Load source and context**

Read `raw/input/{item.filename}` — this is the source document.

Read `wiki/index.md` — existing entity catalog for cross-link identification.

Read all files in `wiki/stakeholders/` to build a known-names registry. For each STK page found, extract both the `name` field and the `nickname` field. Build a lookup: `known_names` = set of all name values AND all non-empty nickname values across all STK pages.

Read `.sara/config.json` — needed by `/sara-add-stakeholder` when invoked inline.

**Step 3 — Generate blocker list**

Using the source document and existing wiki context, identify all blockers in priority order:

Priority 1 — Unknown stakeholders: scan the full source for every person mentioned. For each person found, check if their name or any nickname they are referred to by appears in `known_names`. Collect ALL unknown persons before proceeding. Do not process any Priority 2+ blockers until all unknown persons are listed.

Priority 2 — Ambiguous entity type: identify any passage that could be classified as multiple entity types (REQ vs DEC vs RISK). List each ambiguity.

Priority 3 — Missing context gaps: identify references to concepts, systems, or decisions mentioned in the source that are not present in wiki/index.md and whose meaning is unclear from the source alone.

Priority 4 — Cross-link candidates: identify topics in the source that clearly relate to an existing wiki entity (by matching title or description in wiki/index.md). List each candidate with the wiki entity ID.

Present the full blocker list to the user as a structured summary before resolving anything.

**Step 4 — Resolve unknown stakeholders (Priority 1)**

For each unknown stakeholder identified in Step 3:

  Output: `"Unknown stakeholder: {name}. Resolving via /sara-add-stakeholder."`

  Read `.claude/skills/sara-add-stakeholder/SKILL.md`.
  Execute the sara-add-stakeholder skill inline for this stakeholder.
  Pass `{name}` as the `$ARGUMENTS` value so the name prompt is skipped.
  Capture the returned `{STK-NNN}` ID from the skill's output.
  Add `{name} → {STK-NNN}` to the running discussion_notes context.

After all unknown stakeholders are resolved, proceed to Step 5.

**Step 5 — Work through remaining blockers (Priority 2 → 4)**

For each remaining blocker from the list (process in priority order: 2, then 3, then 4):

  Present the specific blocker to the user as plain text with context from the source document. Example format:
  `"Blocker [entity type ambiguity]: The passage '...' could be classified as REQ or DEC. Which entity type is correct?"`

  Wait for the user's reply. (Plain text wait — do NOT use AskUserQuestion for open-ended clarifications; freeform rule applies.)

  Incorporate the user's response into the running discussion_notes string.

  Mark that blocker resolved. Proceed to the next blocker.

Declare completion ONLY when all blockers (Priority 1 through 4) are resolved and the list is empty.

**Step 6 — Write resolved context and advance stage**

Compile `{discussion_notes}` as a single plain-text string summarising: resolved stakeholders with their STK-NNN IDs, entity type decisions made, context gaps filled, and cross-link confirmations.

Read `.sara/pipeline-state.json`.
Update `items["{N}"]`:
  - `stage`: `"extracting"`
  - `discussion_notes`: `"{discussion_notes}"`
Write the modified JSON back to `.sara/pipeline-state.json` using the Write tool.
Do NOT use Bash jq or sed.

Output:
```
Discussion complete. All blockers resolved.
Discussion notes saved to pipeline-state.json (item {N} stage: extracting).
Run /sara-extract {N} to proceed to extraction.
```

**Notes block:**
- Stakeholder matching must check BOTH `name` AND `nickname` fields in every STK page. A source reference to "Raj" is NOT unknown if any STK page has `nickname: "Raj"` — even if the page name is "Rajiwath Patel".
- Blocker Priority 2–4 resolution uses plain-text output and waits for the user's reply. Do NOT use AskUserQuestion for open-ended questions (freeform rule). AskUserQuestion is used only within the inline sara-add-stakeholder sub-skill.
- Stage advance (`extracting`) happens ONLY after all blockers are resolved. Do not write `extracting` partway through the session.
- The discussion_notes string is the key output — it carries resolved context forward into /sara-extract. Make it specific: include STK-NNN IDs, entity type decisions, wiki entity IDs for confirmed cross-links.
- N argument must be a positive integer string key. `items["1"]` not `items["MTG-001"]`.
