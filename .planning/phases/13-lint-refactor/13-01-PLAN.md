<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| wiki/*.md → sara-lint | Wiki files are read and their content is parsed — a malformed or adversarially crafted wiki page could cause incorrect field inference or YAML parsing errors |
| .sara/config.json → sara-lint | Config file content is used to drive segments inference — a missing or malformed file must degrade gracefully |
| git commit output → sara-lint | The skill checks exit codes from git operations — a git repository in a bad state could cause commit failures that must not halt the full lint run |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Tampering | Write tool on wiki pages | accept | Skill operates on local files under user control; all writes are explicit and individually confirmed by AskUserQuestion |
| T-13-02 | Denial of Service | grep scan on large wiki | accept | grep is read-only and bounded by the wiki directory; no PII, no external network |
| T-13-03 | Information Disclosure | config.json read | accept | config.json contains only project-level segment/department lists; no secrets |
| T-13-04 | Elevation of Privilege | git commit via Bash | mitigate | Commit only stages explicit file paths (never directory globs); commit messages are templated — no user-supplied strings are interpolated into the commit command directly |
| T-13-05 | Repudiation | Per-fix commits | accept | One commit per fix gives a full, independently revertable audit trail — this is a feature, not a risk |
</threat_model>
